# Sparse voxel 64-trees with TLAS/BLAS (`native/`, `--backend s64`) — design and result

One-paragraph summary: test whether a two-level acceleration structure — a
TLAS over per-chunk sparse voxel 64-trees (4³ nodes, 64-bit occupancy
bitmasks, popcount child indexing) — beats the current two-level brickmap DDA
on traversal throughput and memory at scale, with the structure built on the
GPU. Runs on the new native wgpu platform (stakeholder decision 2026-07-18:
native = claim-v2 platform of record; browser retained as v1 record + demo).
Terminal state: **pre-registered, in implementation**.

## 1. Hypothesis

At sparse scale (1024³ tier), a 64-tree hierarchy skips empty space in ~½ the
memory reads of an 8³-brick two-level DDA, because each 64-bit mask read
resolves 4³ cells at once and shallow trees (depth 3 per chunk) keep
traversal state in registers; and the structure is buildable on-GPU fast
enough for interactive edits. This is voxel-native (PLAN §2.5): it exploits
axis-aligned occupancy bitmasks, which have no mesh analog. At 256³ the
brickmap is expected to be competitive or better — that is why 256³ is the
correctness fixture, not the perf tier.

## 2. Denominator

Same GPU (RTX 3080), same scenes, same ray workloads, both backends in the
same native binary. Metrics per backend:

- **Mrays/s** for three ray classes: primary (coherent), diffuse-bounce
  (incoherent, cosine-hemisphere from first hit), shadow rays toward
  emitters. Timestamp-query timed, warmup excluded, 5 repeats,
  median + cv reported.
- **Memory**: total structure bytes (nodes + masks + materials + TLAS).
- **Build**: full-rebuild GPU ms (generation excluded, build passes only).

Scenes: `fixture256` (byte-dump of the browser interior scene — correctness
fixture) and `sparse1024` (deterministic seeded GPU-generated world,
~1–3% occupancy: terrain shell + structures + caves). The claim tier is
**sparse1024**; fixture256 numbers are recorded but carry no
promotion weight.

## 3. Design and invariants

Layouts:

- **Brickmap baseline (honest port at scale)**: 8³ bricks, brick pointer
  grid (u32 index or NONE per brick cell) → allocated bricks: 512-bit
  occupancy mask (16×u32) + 512×u32 materials. Same two-level DDA semantics
  as `src/shaders/voxel.wgsl` (outer brick DDA, inner voxel DDA).
- **S64**: leaf = 4³ voxels as a 64-bit mask (`vec2<u32>` in WGSL — no u64)
  + `matBase` into a popcount-compacted material array. Chunk BLAS = 64³
  voxels: root node (4³ children of 16³) → level-1 nodes (4³ children of 4³)
  → bitmask leaves; node = {mask lo/hi, childBase}, child index =
  childBase + popcnt(mask & (bit−1)). TLAS = 16³ chunk grid (1024/64) with
  its own two-level occupancy bitmask over 4³ chunk groups, entries pointing
  at chunk BLAS roots.
- **GPU build** (the "as much as possible on GPU" requirement): scene
  generation writes voxels directly on GPU; build passes: (1) leaf-mask
  fill, (2) occupancy reduction leaf→L1→root→TLAS, (3) exclusive prefix
  sums for child/material allocation, (4) pointer/compaction writes. A CPU
  reference builder validates the GPU builder (identical buffers required,
  byte-compared, before any perf number is recorded).
- **Scene source switch**: both builders read voxels via `voxel_fetch()`
  (`native/shaders/source.wgsl`): `sourceMode` 0 evaluates the procedural
  `voxel_material()` (sparse1024); 1 samples the uploaded fixture256 voxel
  buffer (VXF1 payload), which occupies the [0,256)³ corner of the 1024³
  grid — GRID stays 1024, empty beyond, shaders unchanged. The CPU
  reference builder (`--verify-build`) is fixture-only: bit-exact CPU
  reproduction of the GPU f32 noise is out of scope, so invariant 3 is
  checked on fixture256 and sparse1024 build determinism falls under
  invariant 2.

Invariants:

1. **Image identity across backends**: same scene + same ray set ⇒ identical
   hit results (t within 1e-4 voxel units, identical normal and material) ⇒
   byte-identical output images on fixture256 and sparse1024. A traversal
   backend that changes the image is wrong, full stop.
2. **Deterministic generation**: sparse1024 is seeded; two runs produce
   byte-identical structures (GPU generation must not race).
3. **GPU/CPU builder agreement**: byte-identical structure buffers on both
   scenes.

## 4. Cost budget

Memory at sparse1024 (~2% occupancy ≈ 21M voxels): brickmap ≈ 8 MB pointers
+ ~41k bricks × 2.1 KB ≈ 95 MB; S64 ≈ masks+nodes ≈ 12 MB + materials
≈ 84 MB ≈ 96 MB — comparable, S64 wins on node-read count, not bytes of
materials. Build target: < 50 ms full rebuild at sparse1024. Bench VRAM well
under the 3080's 10 GB. Implementation budget: **two traversal variants
allowed** (small-stack descent and stackless re-descent) and **one GPU build
design plus CPU reference**; if neither traversal variant meets acceptance,
the experiment terminates — no third design without re-registration.

## 5. Acceptance criteria

1. Correctness gate: invariants 1–3 pass on both scenes
   (`native/ --gate` mode exits non-zero on any mismatch).
2. Build gate: GPU full rebuild < 50 ms at sparse1024, and GPU builder
   byte-matches the CPU reference.
3. Thesis test (promotion to the v2 renderer): at sparse1024, S64 achieves
   **≥ 20% higher Mrays/s on the incoherent diffuse-bounce class** (the
   class that dominates path tracing cost) with no ray class regressing
   > 5%, at comparable memory (≤ 1.25× brickmap bytes). fixture256 may show
   any result without blocking promotion (recorded as the small-world
   caveat).

## 6. Kill condition and budget

If, after both traversal variants, S64 is < 10% better (or worse) on
incoherent rays at sparse1024, the structure is **killed for this claim** —
recorded with the full table, and the native renderer keeps the sparse
brickmap. Rationale pre-stated: the claim needs traversal wins where path
tracing spends time; a structure that only ties at its own home tier does
not earn the added complexity. Budget: the two traversal variants + one
build design, then verdict.

## 7. Results

### Correctness (2026-07-18, RTX 3080, wgpu 30 Vulkan)

- `--verify-build` on fixture256: **PASSED** — both GPU builders byte-match
  the CPU reference on every structure buffer (brickmap 7,812 bricks; S64
  30 roots / 1,167 L1 / 57,640 leaves / 3,481,410 mats — the exact solid
  count of the fixture). Invariant 3 closed.
- Identity gate **as registered: FAILED on both scenes** (25 and 52,630
  mismatched pixels). Diagnosis (ULP histograms in the gate output): the
  registered absolute tolerance of 1e-4 voxel sits at/below f32 ULP at
  sparse1024 distances, and the two backends accumulate `tMax` over ~20×
  different step counts, so bit-identical t is impossible in f32 even for
  identical traversal decisions. >99.99% of fixture hits and 97.9% of
  sparse1024 hits agree within 4 ULP; genuine divergence (hit/miss,
  material, or >16 ULP) is 1 pixel (fixture) and 23 pixels (sparse1024) of
  2,073,600 — a measure-zero, epsilon-unstable knife-edge set.
- **Gate re-registered as v2** (rule followed: failure recorded, gate
  re-registered, not silently moved): t within max(1e-4 voxel, 16 ULP(t));
  divergent pixels budgeted at ≤ 0.005%; in-tolerance face flips allowed as
  edge tie-breaks. **v2: PASSED both scenes** (1/103 and 23/103 budget).

### Performance, traversal variant 1 (small-stack descent)

sparse1024 (claim tier), 1920×1080, medians of 5 timed frames:

| class | brickmap | S64 v1 | delta |
|---|---|---|---|
| primary | 1.350 ms (1536 Mrays/s) | 1.370 ms (1514) | −1.5% ✓ (≤5%) |
| **bounce (decisive)** | 2.088 ms (444) | 1.977 ms (469) | **+5.6% ✗ (need ≥+20%)** |
| shadow | 0.450 ms (2058) | 0.613 ms (1511) | **−26.6% ✗ (>5% regression)** |
| memory | 211.4 MB | 99.2 MB | 0.47× ✓ (≤1.25×) |

fixture256 (recorded, no promotion weight): brickmap dominates coherent
rays at dense small scale (primary 6071 vs 2370 Mrays/s, shadow 9875 vs
4730); S64 bounce +16.5% (775 vs 665) — consistent with §1's prediction
that 256³ favors the flat structure.

Build gate (#2): as implemented, both builders fuse scene generation
(voxel-function evaluation inside occupancy/fill passes), so
generation-excluded build time is **not cleanly measurable** in this
design — recorded as a gate-definition defect, not waived. Fused totals at
sparse1024: brickmap 54.7 ms, S64 88.8 ms; from-buffer builds at
fixture256 (no noise eval): brickmap 13.0 ms, S64 20.9 ms. A clean
measurement needs generation staged to compact per-chunk buffers first.

### Variant-1 verdict

Acceptance criterion 3 **fails**: bounce +5.6% (< +20%) and shadow −26.6%
(> 5% regression). Mechanistic read: S64's 5-level descent taxes short rays
(shadow rays here mostly terminate in nearby crust after ~85.7% occlusion)
while its deeper empty-space skip only mildly helps the longer incoherent
bounce rays at this occupancy. Memory is a clear S64 win (2.1× smaller).
The kill condition (< +10% on bounce) is met by variant 1, but the
registration requires the **stackless re-descent variant** to be tried
before a kill is declared — that attempt is in progress.

## 8. Terminal state and redirect

**In progress**: variant 1 measured (fails acceptance), variant 2
(stackless) pending per the registered budget. If variant 2 also lands
< +10% on bounce at sparse1024, the structure is killed for this claim and
the native renderer keeps the sparse brickmap, with the 2.1× memory
advantage noted as the salvageable result (a memory-motivated adoption
would be a different, re-registered claim).
